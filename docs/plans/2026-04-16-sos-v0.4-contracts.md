# SOS v0.4 — "Contracts" — Freeze the interfaces so language is replaceable

**Date:** 2026-04-16
**Author:** sos-dev (connectivity lead)
**Target:** v0.4.0 release
**Scope:** software-maturity discipline, not new features
**Follows:** v0.3.0 (current)
**Depends on:** nothing — this is the bottom of the stack

## Why this release exists

SOS is at v0.3.0. The Python implementation works, ships customers, handles real traffic. But the interfaces between services are encoded in Python type hints and ad-hoc JSON shapes. That makes three things hard:

1. **Traceability** — every service speaks slightly different JSON. Errors are strings, fields are optional-in-practice, identity is implicit. Today's accounting ledger attributes 3,470 calls to "kasra" because the source field isn't structurally required.
2. **Multi-language** — you can't port a service to another language (Rust is the stated future target) when the contract only exists as Python type annotations.
3. **Change safety** — the CHANGELOG has been silent since 0.1.1 while the code has moved four minor versions. Breaking changes between 0.x minors are allowed by SemVer but currently silent in practice.

v0.4 closes all three with published, versioned, testable contracts. It is **pure infrastructure work** — no new customer-facing features — but it is the bottom of every other plan (squad-as-living-graph, Inkwell 4.1 Glass, $MIND mainnet). Everything downstream is fragile until v0.4 ships.

## The five deliverables

### 1. Agent Card schema (JSON Schema, versioned, published)

Formalize the ad-hoc Redis hash I used to register kasra + trop today. Every entry in `sos:registry:{agent}` conforms to this schema, every MCP `peers` response returns records that parse against it.

- File: `sos/contracts/schemas/agent_card_v1.json`
- Validator: `sos/contracts/agent_card.py` — Pydantic model with `from_redis_hash()` / `to_redis_hash()`
- Integration: `sos/bus/announce.sh` (or its Python equivalent) writes schema-conformant records only
- MCP tool upgrade: `mcp__sos__peers` returns a list of Agent Cards instead of raw HSET dumps
- Version field on every card (`agent_card_version: "1.0"`)

### 2. Message schema registry (every bus message type has a schema)

Every `XADD` to `sos:stream:*` validates against a schema. Fields that can't be optional aren't optional. `source`, `target`, `timestamp`, `version`, `message_id` become structurally required — end of the flat-identity bug by construction.

- Directory: `sos/contracts/schemas/messages/`
- Types seeded: `announce.v1.json`, `send.v1.json`, `wake.v1.json`, `task_created.v1.json`, `task_claimed.v1.json`, `task_completed.v1.json`, `agent_joined.v1.json`
- Validator: `sos/contracts/messages.py` — dispatches schema by `type` field
- Enforcement: `sos/bus/core.py` + `sos/services/bus/delivery.py` validate on send, drop-and-log on violation

### 3. OpenAPI specs per HTTP service

Every service running on a port gets a `openapi.yaml` declaring its endpoints, request/response shapes, error codes, auth requirements. Generated from FastAPI where possible; hand-written for the Rust-destined hot paths.

- Services covered: `squad :8060`, `mirror :8844`, `saas :8075`, `dashboard :8090`, `bus gateway :6070`, `engine :6060`, `memory :6061`, `content :8020`
- Files: `sos/contracts/openapi/<service>.yaml`
- CI check: every endpoint in code must be declared in the spec; deviation fails the build
- Published: FastAPI serves `/openapi.json` at each service root; static YAML shipped in the repo for non-FastAPI services

### 4. Contract tests (cross-service, not just per-endpoint)

For every schema and OpenAPI spec, a test that exercises the real service with real fixtures and asserts the response validates. These are what a future Rust port would target — if the Rust service passes the same contract tests, it's a drop-in.

- Location: `tests/contracts/`
- Covers: Agent Card round-trip, each bus message type, each HTTP endpoint
- Runs: CI on every PR, locally via `uv run pytest tests/contracts/`

### 5. Error taxonomy (enum, not free-text)

Every error that crosses a service boundary has a stable machine-readable code. HTTP responses include `{"code": "SOS-1042", "message": "...", "details": {...}}`. `sos/contracts/errors.py` already exists — formalize it.

- Code ranges (already declared in CHANGELOG 0.1.0): 1xxx auth, 2xxx routing, 3xxx task, 4xxx wallet, 5xxx fmaap, 6xxx mirror, 7xxx tenant, 8xxx platform
- Registry: `sos/contracts/errors.py` as single source
- Tests: exhaustive list — no raising an unregistered code
- Docs: `sos/contracts/errors.md` — table of every code, cause, remediation

## Concrete side-effects (why this is not "just process")

Landing v0.4 fixes these real problems directly:

| Current bug | Fixed by | How |
|---|---|---|
| Flat identity (all bus messages tagged `agent:kasra`) | Deliverable 2 | `source` field structurally required, typed to a registered agent |
| Accounting ledger mis-attributes | Deliverable 1 + 2 | Agent Cards + message schema force per-agent attribution |
| Dashboard `_fetch_memory` called `/engrams` that didn't exist | Deliverable 3 | OpenAPI spec catches this at build time |
| SEC-001 regression (`entry["token"] == token` against empty raw field) | Deliverable 3 | OpenAPI spec for auth would require `token_hash` field; reviewers see it immediately |
| nginx `301 → mumega.com` instead of proxy | Deliverable 3 | Contract test for `app.mumega.com` expects 200 on `/login`, not 301 elsewhere |
| sos-claude remote connector flat | Deliverable 1 + 2 | per-agent Cards + required `source` means every claude.ai identity is distinct at the protocol level |

Each of those incidents had a root cause that a contract layer would have caught. The contract layer is therefore **the medic's favorite deliverable** — the backlog shrinks significantly once it lands.

## What this release explicitly does NOT do

- Does not build FMAAP real engine (that's v0.5 or parallel)
- Does not add new tools, adapters, or integrations
- Does not touch the billing pipeline
- Does not attempt the Rust port (that's post-1.0)
- Does not break wire-compatibility — all existing endpoints keep working, just gain spec documents + validation

## Definition of done for v0.4.0

- Every agent in `sos:registry:*` has a schema-valid Agent Card
- Every bus message has a `source` that resolves to a registered Agent Card
- Every HTTP service has an OpenAPI document served at its root
- Every service has a contract test that runs in CI
- CHANGELOG.md matches git history back to 0.1.1
- `v0.4.0` tag exists, release notes published

After v0.4, the backlog task "Fix sos-claude MCP identity flatness" closes by construction — it cannot exist in a world where messages fail to validate without a real `source` field.

## What this unlocks

v0.5 "Traceable" — OpenTelemetry becomes mechanical once every message has a stable ID and every endpoint has a declared contract.

v0.6 "Migratable" — Alembic can hang off the contract schemas for storage types.

v0.7 "Isolated" — Adapters become real services with declared interfaces.

v0.8 "Capable" — Capability auth enforces against declared resources, not ad-hoc strings.

v0.9 "Frozen" — contract freeze. No more breaking changes until 2.0.

v1.0.0 — ship. Then the Rust port is mechanical: re-implement the contracts, pass the tests, one service at a time.
