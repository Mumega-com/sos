## 1 — Approach

**Chosen: Phased-monolithic.** One v0.4.7 sprint, one tag at the end, one commit per phase so each boundary is clean. If a phase proves too heavy, stop at the phase boundary; the already-landed phases stay green because each shrinks the ignore list incrementally.

Alternative (not taken): v0.4.7-alpha.1 partial closure. Rejected because Hadi prefers bundled PRs over churn (saved feedback: "the single bundled PR was the right call").

## 3 — Risks

- **`_lookup_token`'s actual logic.** If it does more than DB lookup by token (e.g. cross-checks bus tokens, computes scopes), the Phase 1.3 env-only swap is insufficient. Mitigation: read the function before Step 1.3; if non-trivial, add `/auth/verify` endpoint to squad app and call it via SquadClient.
- **In-process Marketplace state.** `_marketplace = Marketplace()` at module load suggests possible cached state. Mitigation: inspect Marketplace class — if it's stateless (just wraps a DB), HTTP is fine.
- **Billing webhook routing.** Stripe dashboard needs to point at billing service directly after Phase 3. If the org's Stripe config points at MCP today, Phase 3 is a runbook change, not just code. Mitigation: document in `docs/billing-webhook-routing.md`; flag the production cutover separately from the code ship.
- **Async audit fire-and-forget.** `log_tool_call` must not block tool dispatch. Mitigation: use AsyncSaasClient + create_task / fire-and-forget; accept that logs may be lost on crash (current behavior is also best-effort).
- **Session time budget.** 4 phases + release = large. If phase 2 or 3 proves messier than predicted, valid stopping points are after any phase boundary. Partial shipping means "R2 down from 5 to N"; the sprint can resume next session.
