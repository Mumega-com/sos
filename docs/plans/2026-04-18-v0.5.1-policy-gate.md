# v0.5.1 — Unified Policy Gate + POC Migration

**Date:** 2026-04-18
**Predecessor:** v0.5.0 (R0 floor lock + audit stream) — shipped 2026-04-18, commit `9f90ed50`.
**Scope:** Close the "permission check scatter" by introducing a single
`can_execute()` gate in the kernel and proving it with one service migration.
**Deferred to v0.5.2:** Arbitration (conflict resolution between agents).

## Why now

v0.5.0 gave us a floor (R0 lock) and an audit spine (`kernel.audit`). The
reconnaissance before this sprint found **11 scattered permission sites** —
services each invent their own combination of bearer-scope checks, admin
role checks, FMAAP calls, and capability verification. Every new service
becomes a new chance to drift.

The fix is to compose all five signals in one place and have callers ask
exactly one question: *may this agent perform this action on this resource?*

Durability matters: same discipline as v0.5.0, design the interface well
enough that we don't reopen it when policy tiers or capability scopes grow.

## Architecture

```
caller                          sos.kernel.policy.gate
  │                              │
  │  can_execute(agent, action,  │
  │   resource, context)         │
  ├────────────────────────────▶ │
  │                              ├─▶ verify_bearer            (sos.kernel.auth)
  │                              ├─▶ tenant_scope check       (inline)
  │                              ├─▶ capability verification  (capability_auth)
  │                              ├─▶ FMAAP 5-pillar validate  (fmaap)
  │                              ├─▶ governance tier lookup   (governance)
  │                              ├─▶ audit: POLICY_DECISION   (kernel.audit)
  │                              │
  │  PolicyDecision(allow=True,  │
  │   reason="…", tier="…")      │
  │ ◀────────────────────────────┤
```

One gate. One contract. Every signal composable. Audit written once per call.

## Contract — frozen for durability

`sos.contracts.policy.PolicyDecision` (Pydantic v2 frozen model):

```python
class PolicyDecision(BaseModel):
    model_config = ConfigDict(frozen=True)

    allowed: bool
    reason: str
    tier: str                              # governance tier: act_freely | batch_approve | human_gate | dual_approval | denied
    agent: str | None = None
    tenant: str | None = None
    action: str
    resource: str
    pillars_passed: list[str] = []         # which FMAAP pillars passed (for durability — additive)
    pillars_failed: list[str] = []
    capability_ok: bool | None = None      # None = not required by env, True/False = checked
    audit_id: str | None = None            # ID of the POLICY_DECISION event we wrote
    metadata: dict[str, Any] = {}
```

Schema stability test (`tests/contracts/test_policy_schema_stable.py`) snapshots
these fields from day one so future kernel iterations add fields only.

## Gate surface

`sos.kernel.policy.gate`:

```python
async def can_execute(
    *,
    agent: str,
    action: str,
    resource: str,
    tenant: str = "mumega",
    authorization: str | None = None,     # raw Bearer header if HTTP caller
    capability: CapabilityModel | None = None,
    context: dict[str, Any] | None = None,
) -> PolicyDecision: ...
```

That's it. No class, no config object. New signals added by composing
them inside `can_execute` — not by adding new public functions.

## Durability check — what makes this robust

- **Extensible contract:** `pillars_passed` / `pillars_failed` as lists, `metadata` as free-form dict. Add new FMAAP pillars without changing the shape.
- **Additive-only decision fields:** schema stability test enforces it.
- **Composition, not plugins:** no policy plugin registry that another engineer has to learn. New checks are 5 new lines inside `can_execute`.
- **Audit-native:** every decision emits an event. Dashboards, alerts, policy review all read the same stream.
- **Fail-open for availability, fail-closed for security:** FMAAP DB unavailable → warn + allow (availability); bearer absent → deny (security). Documented inline.

## Risk

- **FMAAP engine reads SQLite.** If the DB is missing (fresh tenant) the engine returns `valid=False` for every pillar. Gate must handle this as "soft allow + warning" rather than hard deny — else every fresh tenant action is denied. Addressed in Step 4.
- **Async vs sync.** Gate is async; FMAAP engine is sync. Call engine directly from async (cheap, no I/O beyond sqlite local read) rather than threading an executor.
- **`before_action` caller contract.** No external caller exists today (confirmed v0.5.0 recon). Still, return shape stays compatible: `{"allowed": bool, "tier": str, "intent_id": str}`.
