# 2026-04-18 ¬∑ v0.5.0 prep ‚Äî close the `kernel ‚Üí services.economy.metabolism` leak

## The one remaining kernel‚Üíservices leak

```
sos/kernel/governance.py:102:  from sos.services.economy.metabolism import can_spend
```

That's it. A single inline import inside `before_action()` that
consults budget before allowing a governed action. It's currently
wrapped in `except ImportError: pass` (silently allows the action
if metabolism isn't importable), but `ImportError` is the wrong
signal ‚Äî the module is always importable; the real failure mode is
runtime misconfiguration (missing DB, bad params).

## Why kernel must not import services

- **R1** forbids services from importing services. R2 forbids
  clients/adapters/cli/agents from importing service internals.
  There is no "R0" yet, but the **kernel** is the sternest of these:
  it's the shared, dependency-free floor every service depends on.
  If `sos.kernel.governance` imports `sos.services.economy.metabolism`,
  then every service that imports governance transitively pulls in
  economy, and suddenly "kernel" is not a floor ‚Äî it's a ceiling.
- The v0.4.7 MCP sweep proved the pattern: reach-ins become HTTP
  calls through `sos.clients.*`. Kernel gets the same treatment.

## Current shape

```python
# sos/kernel/governance.py ~100-122
try:
    from sos.services.economy.metabolism import can_spend
    budget_check = can_spend(project, estimated_cost)
    if not budget_check["allowed"]:
        ...
except ImportError:
    pass  # metabolism module optional
```

`can_spend(project, cost) -> dict` returns
`{allowed, budget, spent, remaining, pct_used, reason, warning?}`.
Lives in `sos/services/economy/metabolism.py`. Reads SQLite at
`~/.mumega/metabolism.db`. Synchronous.

## Options

### Option A ‚Äî Expose `can_spend` on economy's HTTP surface, call via `EconomyClient`
**What:** Add `GET /budget/can-spend?project=X&cost=Y` to
`sos/services/economy/app.py` that delegates to
`metabolism.can_spend(...)` and returns the dict unchanged. Add
`EconomyClient.can_spend(project, cost=0.0) -> dict` (sync + async).
Governance becomes:
```python
from sos.clients.economy import EconomyClient
_economy_client = EconomyClient()  # module-level
...
try:
    budget_check = _economy_client.can_spend(project, estimated_cost)
    ...
except Exception as exc:
    logger.debug("budget check unavailable: %s", exc)
    # fail-open ‚Äî preserve current ImportError-swallowing behavior
```
**Tradeoffs:** Mirrors the v0.4.7 pattern exactly. Kernel now imports
`sos.clients.economy` instead of `sos.services.economy.metabolism`
‚Äî which is R2-clean because clients are the sanctioned boundary
crossing. Sync HTTP adds ~1-5ms to every governed action; governance
is already a hot path so this matters, but not catastrophically.
Economy service must be up for governance to consult budget ‚Äî
fail-open preserves availability.

### Option B ‚Äî Move `can_spend` into kernel
**What:** Split metabolism: the synchronous budget-check function
moves to `sos/kernel/budget.py` (reads the same SQLite); the revenue
tracking, Stripe webhook, digest cycle, and allocation all stay in
economy service. Governance imports from kernel.
**Tradeoffs:** No HTTP hop ‚Üí no latency cost. But kernel now owns
SQLite schema + DB path + allocation algorithm ‚Äî that's business
logic bleeding into the floor. And the SQLite file is owned by
economy; if both kernel and economy read+write it, we have two
writers without coordination. Rejected unless we also move the DB
ownership, which is a big lift.

### Option C ‚Äî Event-driven (fire-and-forget)
**What:** Governance emits `budget.check_requested` on the bus and
allows the action immediately. Economy subscribes and later emits
`budget.breach` alerts. No synchronous gate.
**Tradeoffs:** Scales, removes kernel‚Üíeconomy coupling entirely.
But loses the pre-action gate ‚Äî governance becomes observational,
not enforcing. Budget blocks disappear. Rejected; this is a
separate architectural choice ("soft budgets") we haven't made.

### Option D ‚Äî Drop the check
**What:** Remove the `can_spend` block from governance. Budget is
enforced elsewhere (per-adapter ceilings in `sos.adapters.router`,
which already use `EconomyClient.get_wallet()`).
**Tradeoffs:** Zero coupling. But the `before_action()` check is
coarser-grained than adapter ceilings ‚Äî it catches non-LLM actions
too (tool calls, external API hits). Losing it is a real behavior
change. Rejected unless product-side agrees budgets are adapter-only.

## Recommendation

**Option A.** Matches the v0.4.7 playbook, preserves behavior,
keeps economy as the single source of budget truth. The ~1-5ms
overhead is acceptable for a gate that already runs async code
and logs intents.

## Out of scope for v0.5.0

- Rewriting metabolism as a FastAPI service of its own. It stays
  inside `sos.services.economy` as a library that the economy
  service exposes over HTTP.
- The three `sos.cli -> sos.services.*.__main__` R2 ignores ‚Äî those
  are the legitimate packaging pattern; they go away when v0.5
  splits `sos.cli` into its own sibling package.
- `sos.adapters.telegram -> sos.services.economy.metabolism` ‚Äî same
  shape as governance, same fix: swap to `EconomyClient`. Ship
  with Option A in the same v0.5 wave.
