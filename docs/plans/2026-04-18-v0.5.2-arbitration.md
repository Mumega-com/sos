# v0.5.2 — Arbitration: Intent → Proposal → Ratification

**Date:** 2026-04-18
**Predecessors:** v0.5.0 (floor + audit), v0.5.1 (unified gate).
**Scope:** Deliberative conflict resolution between agents acting on the
same resource. Closes the third leg of the "kernel enforces the blueprint"
triplet.
**Design (selected):** Intent-proposal-ratification (Option C).

## Why now

v0.5.0 audited. v0.5.1 gated each call. But two agents calling the gate
independently still collide — Loom says publish, Athena says hold, and
the gate allows both because it never sees the other. Arbitration is the
layer that looks across proposers and picks a winner.

## The architectural insight

Proposals ARE already `AuditEventKind.INTENT` events. They land on disk,
sortable by timestamp, indexed by (tenant, action, target). Arbitration
is a **read-over-audit + decision function** — no new storage, no new
stream, no new durability boundary to get right. The audit spine pays
off exactly as we designed it.

```
agent.propose_intent(...)  ─▶  append INTENT event (existing audit)
                                        │
                                        ▼
arbitrate(resource, window_ms)  ─▶  read_events(kind=INTENT,
                                                filter target == resource,
                                                filter timestamp >= now - window)
                                        │
                                        ▼
           pick winner by (priority → coherence → recency)
                                        │
                                        ▼
             append ARBITRATION event     (new kind, already reserved)
                                        │
                                        ▼
         return ArbitrationDecision (frozen contract)
```

Nothing new to persist. The INTENT kind was reserved for this in
v0.5.0 — the receipt is being cashed.

## Contract

`sos.contracts.arbitration.ArbitrationDecision` (Pydantic v2 frozen):

```python
class ArbitrationDecision(BaseModel):
    model_config = ConfigDict(frozen=True)

    resource: str
    tenant: str
    strategy: str                          # "priority+coherence+recency" (v0.5.2 baseline)
    window_ms: int
    winner_agent: str | None               # None when no proposals in window
    winner_proposal_id: str | None
    winner_reason: str
    losers: list[LoserRecord] = []         # structured per-loser denial
    proposal_count: int
    audit_id: str | None = None
    metadata: dict[str, Any] = {}


class LoserRecord(BaseModel):
    model_config = ConfigDict(frozen=True)

    agent: str
    proposal_id: str
    reason: str
    priority: int | None = None
```

Schema stability test locks the v0.5.2 baseline — additive-only.

## Surface

`sos.kernel.arbitration`:

```python
async def propose_intent(
    *,
    agent: str,
    action: str,
    resource: str,
    tenant: str = "mumega",
    priority: int = 0,                # higher wins
    metadata: dict[str, Any] | None = None,
) -> str: ...                          # returns proposal_id (= audit event id)

async def arbitrate(
    *,
    resource: str,
    tenant: str = "mumega",
    window_ms: int = 500,
    strategy: str = "priority+coherence+recency",
) -> ArbitrationDecision: ...

def read_proposals(
    tenant: str,
    resource: str,
    *,
    window_ms: int = 500,
) -> list[AuditEvent]: ...              # helper for observability
```

Three public functions. Nothing else.

## Winner rule — "priority + coherence + recency"

1. **Priority tier (hard sort)** — highest `metadata.priority` wins
   outright. Priority 0 is default. Agents that set priority MUST justify
   it (audit trail shows the claim).
2. **Coherence tie-break** — if priorities tie, whichever agent belongs
   to a squad with higher coherence wins. Coherence lookup uses
   `sos.kernel.conductance` (the same matrix FMAAP pillar 1 reads).
3. **Recency tie-break** — if still tied, most recent proposal wins.
   This prefers responsive agents over slow ones, and the timestamp is
   already on every INTENT event.

Strategy name recorded in the ArbitrationDecision. Future strategies
("fmaap-weighted", "governance-tier-aware") ship as new string values
without schema churn.

## Integration with the gate

The gate gets a new keyword: `propose_first: bool = False`.

```python
await can_execute(
    agent="loom",
    action="content_publish",
    resource="post_42",
    tenant="mumega",
    propose_first=True,           # new in v0.5.2
    priority=0,
    window_ms=500,
)
```

When `propose_first=True`:
- Gate calls `propose_intent(...)` first.
- Then calls `arbitrate(resource, tenant, window_ms)`.
- If this caller is the winner → continue through normal gate signals.
- If this caller is a loser → return denial with arbitration reason.

Defaults are `False` — every existing caller still works unchanged.
Callers opt in when they know they're in a contested action.

## Success criteria

1. Two agents proposing on the same resource produce exactly one winner.
2. Losers get structured denial with `LoserRecord.reason` explaining why.
3. Every arbitration writes one `ARBITRATION` audit event.
4. `ArbitrationDecision` schema locked at v0.5.2 baseline.
5. Legacy `~/.sos/governance/intents/` shim removed — audit is the only
   write path for intents.
6. No other service broken.

## Durability — why this doesn't reopen the kernel

- **Contract frozen on day one** — same rule as AuditEvent and PolicyDecision.
- **No new persistence layer** — arbitration reads from audit, writes to audit.
- **Strategy is a string** — new strategies ship as new values, not new code paths.
- **Gate integration is additive** — `propose_first=False` default keeps every
  existing caller working.

When the next three kernel modules (trust signals, capability delegation,
coherence decay) arrive, they add *signals* to the gate and *strategies*
to arbitration. Neither reopens these files.

## Deferred to v0.5.3+

- Per-squad arbitration windows (dynamic, based on coherence).
- Arbitration replay tooling (`sos.cli arbitration replay`) — read ARBITRATION
  events and explain the outcome. Useful but not core.
- Migrating remaining ~10 service-side permission checks to `can_execute`
  (same mop-up deferred from v0.5.1).
