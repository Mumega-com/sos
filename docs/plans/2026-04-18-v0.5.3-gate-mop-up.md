# v0.5.3 — Policy Gate Mop-Up (Wave 1)

**Date:** 2026-04-18
**Predecessors:** v0.5.1 (`can_execute()` gate), v0.5.2 (arbitration).
**Scope:** Migrate the 5 small-and-mechanical service-side permission
sites to `sos.kernel.policy.gate.can_execute()`. Each one was listed as
"deferred v0.5.1 mop-up" in the v0.5.1 CHANGELOG.

## Why now

v0.5.1 shipped `can_execute()` and migrated `sos/services/integrations/app.py`
as proof-of-concept. Every other service still re-implements bearer
verification, tenant scope enforcement, and admin-only guards inline.
That means: different 401/403 error shapes, uneven audit trails, and no
single place to add new signals. Wave 1 closes the gap for the 5 small
services. `saas` (medium) lands in v0.5.4; `squad` (large, needs a
capability→gate bridge) in v0.5.5.

## Target services (wave 1)

| Service | Routes | Auth pattern | Est. LOC |
|---------|--------|--------------|----------|
| economy | 4 | bearer + tenant scope + 1 admin | ~25 |
| registry | 2 | bearer + project scope | ~20 |
| identity | 2 | bearer + scope-gated (non-tenant) | ~20 |
| journeys | 4 | bearer + system/admin only | ~15 |

Total: ~95 LOC, 15 routes, all parallelizable.

## The migration pattern

From `sos/services/integrations/app.py` (v0.5.1 POC):

```python
# Before: inline helpers
token = _verify_bearer(request)
_check_tenant_scope(token, tenant)
if not token.is_admin: raise 403

# After: one gate call + one handoff
decision = await can_execute(
    agent=agent or "anonymous",
    action="integrations:oauth_callback",
    resource=tenant,
    tenant=tenant,
    authorization=request.headers.get("authorization"),
)
_raise_on_deny(decision, require_system=False)
```

`_raise_on_deny(decision, require_system=False)` maps:
- `decision.allowed is False` → `HTTPException(401 or 403, decision.reason)`
- `require_system=True` + allow-reason doesn't mention "system/admin" → `403`

For admin-only routes (`journeys`, `operations`), pass `require_system=True`
to the handoff.
