## Part 2 — Unified Audit Stream (Steps 9-14)

### Why audit now

Issue #123 asks for three new kernel subsystems: policy engine,
audit stream, arbitration. Audit is the **least controversial** —
every service already logs, we just haven't unified the schema or
the sink. Policy and arbitration need design work we're not ready
for tonight. Audit is shippable on its own AND it's the thing
policy+arbitration will write into once they land — so doing audit
first locks the schema other modules bind to.

### The durability principle

The audit contract goes in `sos/contracts/audit.py` once. Every
module binds to that shape. We don't want to change the schema in
v0.5.1 — so the schema below is designed to accommodate policy
decisions and arbitration outcomes **without needing new fields**.

### Contract — `sos/contracts/audit.py`

```python
class AuditEventKind(str, Enum):
    INTENT = "intent"                     # governance.before_action
    POLICY_DECISION = "policy_decision"   # kernel.policy.can_execute (v0.5.1)
    ACTION_COMPLETED = "action_completed" # successful execution
    ACTION_FAILED = "action_failed"       # failed execution
    ARBITRATION = "arbitration"           # conflict resolution (v0.5.2)

class AuditDecision(str, Enum):
    ALLOW = "allow"
    DENY = "deny"
    REQUIRE_APPROVAL = "require_approval"
    NOT_APPLICABLE = "n/a"  # for ACTION_COMPLETED/FAILED where no decision was made

class AuditEvent(BaseModel):
    # Identity
    id: str                                  # ULID for sortable uniqueness
    timestamp: str                           # ISO 8601 UTC

    # Actor
    agent: str                               # acting agent or service name
    tenant: str                              # scope
    trace_id: str | None = None              # cross-service correlation
    parent_event_id: str | None = None       # causal chains (replay)

    # Kind + target
    kind: AuditEventKind
    action: str                              # e.g. "tool.call", "payment.create"
    target: str                              # what was acted on

    # Decision (for INTENT, POLICY_DECISION, ARBITRATION)
    decision: AuditDecision = AuditDecision.NOT_APPLICABLE
    reason: str = ""
    policy_tier: str | None = None           # act_freely/batch_approve/human_gate/dual_approval

    # Economics (linkage to cost tracking)
    cost_micros: int = 0
    cost_currency: str = "USD"

    # Payload (bounded)
    inputs: dict[str, Any] = Field(default_factory=dict)
    outputs: dict[str, Any] = Field(default_factory=dict)
    metadata: dict[str, Any] = Field(default_factory=dict)

    model_config = ConfigDict(frozen=True)   # immutable after creation
```

### Writer — `sos/kernel/audit.py`

**Design: disk is authoritative, bus is observational** (option A3).

- Disk write: `~/.sos/audit/{tenant}/{YYYY-MM-DD}.jsonl` — synchronous,
  fsync on each append. Ordered by write time. Source of truth.
- Bus emit: `XADD sos:audit:{tenant} * …` — async best-effort.
  Real-time consumers (dashboard, alerting) tail this. If Redis is
  down, audit still works; we just lose the real-time feed.

**Public API (small and stable):**

```python
# write path
async def append_event(event: AuditEvent) -> str: ...  # returns id

# read path (replay)
def read_events(
    tenant: str,
    *,
    date: str | None = None,            # ISO date
    kind: AuditEventKind | None = None, # filter
    limit: int = 100,
) -> list[AuditEvent]: ...
```

That's the entire surface. No class, no config object, no middleware
hooks. Services pass an `AuditEvent` in, get an id out. Durable
because the interface is minimal.

### Integration point — rewire governance

`before_action()` currently writes to `~/.sos/governance/intents/{tenant}/{date}.jsonl`.
That IS audit — it's just living in the wrong module. In this sprint
we:

1. Keep `intent` as a kind in the audit event taxonomy.
2. Rewrite `governance.before_action()` to call `audit.append_event(kind=INTENT, ...)`.
3. Keep the legacy `governance/intents/` path as a compatibility shim
   for one version — `audit.py` writes to both paths, governance
   readers keep working. **Plan to remove the shim in v0.5.2.**

Other services (tool calls, bus messages, economy) are NOT rewired in
this sprint. Policy engine + arbitration will be first-class audit
writers in v0.5.1/v0.5.2. For now the contract is stable and ready.

## Risks + mitigations

- **Risk:** Audit disk write adds latency to governance hot path.
  **Mitigation:** JSONL append + fsync is ~1-2ms; governance already
  does this for intents. No regression.
- **Risk:** `EconomyClient.can_spend` HTTP hop adds 1-5ms to every
  governed action.
  **Mitigation:** fail-open preserves availability; measured and
  acceptable for a gate that already runs async code.
- **Risk:** Audit schema changes in v0.5.1 break old events.
  **Mitigation:** `test_audit_schema_stable.py` snapshots the schema;
  v0.5.1 can only *add* optional fields, never remove or rename.
- **Risk:** Redis down → bus emit fails → audit silently loses
  real-time signal.
  **Mitigation:** disk is authoritative; bus is labeled
  "observational" in the doc. Operators understand.

## The durability check

Before shipping, ask: "Do we have to reopen the kernel to add policy
or arbitration in v0.5.1/v0.5.2?"

- Policy will add its decisions as `AuditEvent(kind=POLICY_DECISION)`
  — schema already accommodates this. ✅
- Arbitration will add its outcomes as
  `AuditEvent(kind=ARBITRATION)` — schema already accommodates. ✅
- Policy gate will live in `kernel/policy.py` — new file, doesn't
  touch audit.py. ✅
- Arbitration will live in `kernel/arbitration.py` — new file,
  doesn't touch audit.py. ✅

We reopen the kernel to ADD modules, not to MODIFY existing ones.
That is the durability property we're buying with this design.
