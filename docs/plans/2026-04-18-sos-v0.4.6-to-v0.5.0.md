## 1 — Sequencing: three options

**Option A: Risk-serial (v0.4.6 → v0.4.7 → v0.4.8 → v0.5.0)**
- What: fix R2 ignores first (small), then MCP (big), then hygiene, then kernel.
- Tradeoff: predictable progress on contract closure — vs — v0.4.7 (biggest refactor) done on a cluttered repo; hygiene waits longest.

**Option B: Ease-serial (v0.4.8 → v0.4.6 → v0.4.7 → v0.5.0)** ← **recommended**
- What: hygiene first (pure git mv, no behavior change), then R2 clients, then MCP rewrite, then kernel.
- Tradeoff: clean tree makes the MCP rewrite's import churn much easier — vs — first sprint closes zero contracts (still "passing" though, just no shrink).

**Option C: Parallel (v0.4.8 alongside v0.4.6)**
- What: hygiene branch + client-refactor branch in parallel, then v0.4.7, then v0.5.0.
- Tradeoff: max throughput — vs — merge-conflict window because both touch `scripts/`, `cli/`, `agents/` paths.

**Decision: Option B.** Reasons:
1. v0.4.8 removes 10k+ data files from the package — every subsequent grep/graph-update runs faster.
2. `clients/saas.py` and `clients/operations.py` battle-tested by v0.4.6 before MCP depends on them in v0.4.7.
3. Serial = zero merge-conflict risk.
4. v0.5.0 is the tentpole; having 6,7,8 landed on a clean tree lets it be the focus, not fighting regressions.

If user prefers a different order, sprint steps below are independent and can be re-sequenced.

### Risks

- **Hidden state.** Some service imports exist because the MCP needed in-process access to something that doesn't have an HTTP surface (e.g. an in-memory cache). Mitigation: Step 1 flags these. Worst case: add the HTTP endpoint as prerequisite work.
- **Performance.** 40 tools × HTTP hop = more latency. Mitigation: MCP and services typically co-located; loopback HTTP is <1ms. If it matters, add a unix-socket transport later.
- **Auth boundary.** MCP's bearer auth is currently tangled with squad's auth module (SYSTEM_TOKEN reuse). Step 4 forces a clean kernel boundary — may surface bugs in token resolution. Mitigation: test each tier (system token, project token, customer token) explicitly.
