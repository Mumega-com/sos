## The squad (form once, use sprint)

Five members. Define at sprint start. Reuse across every week. Same squad carries into v0.4.1 (Provider Matrix) and v0.4.2 (Observability Plane) because the work shape repeats (new schemas, new models, new OpenAPIs, new tests).

| Member | Type | Model | Location | Role |
|---|---|---|---|---|
| **sos-schema-author** | stateless subagent | sonnet | `.claude/agents/sos-schema-author.md` | Writes JSON Schema files, one at a time. Reads the spec I hand over + existing `agent_card_v1.json` as pattern. Outputs schema file + one-line summary. |
| **sos-pydantic-author** | stateless subagent | sonnet | `.claude/agents/sos-pydantic-author.md` | Given a JSON Schema, writes the matching Pydantic v2 model with `to_redis_hash()` / `from_redis_hash()` or equivalent round-trip helpers. |
| **sos-openapi-author** | stateless subagent | sonnet | `.claude/agents/sos-openapi-author.md` | Given a FastAPI service file, emits OpenAPI YAML matching existing endpoints + documents error codes + response types. |
| **sos-contract-tester** | stateless subagent | **haiku** | `.claude/agents/sos-contract-tester.md` | Given a schema + model, generates contract tests in `tests/contracts/` that cover round-trip, boundary conditions, invalid-input rejection. Haiku because test-writing is pattern-replication. |
| **sos-medic** | **stateful** | sonnet | `sos/agents/sos-medic/` (tmux session) | On-call responder. Handles any pipe breakage during sprint (wake-daemon, auth, nginx, etc.). Already built today. Persistent EXPERIENCE.md, reads on every new session. |

Coordinator: me (sos-dev, opus, this session).

## Why this shape

- **No-context stateless subagents** (your constraint) — each call briefs the subagent fully in the prompt. No assumption the next call remembers the last. That's how the `Agent` tool already works.
- **Stateful sos-medic** (your constraint) — one persistent Sonnet agent for ops work so it accumulates experience across incidents via EXPERIENCE.md. Already shipped today.
- **Haiku for tests** — test files are pattern-heavy. Haiku at $1/$5 per MTok vs Sonnet at $3/$15 = 67-70% cost savings on a high-volume subagent. Schema-author, OpenAPI-author need Sonnet judgment; tester doesn't.
- **Form once, use sprint** (your constraint) — four specs written Week 1, reused every week of v0.4.0 + both follow-on releases. No subagent creation churn.

## Dispatch protocol (how I use them each week)

Every deliverable follows the same pipeline. I dispatch in parallel when dependencies allow.

```
For each new contract (schema or OpenAPI):

  ┌─────────── me ───────────┐
  │ describe the deliverable │
  │ in a Task tool brief     │
  └──────────┬───────────────┘
             │
     parallel dispatch (when possible):
     │
     ├──► sos-schema-author (Sonnet) ──► produces JSON Schema
     │                                    │
     │                                    ▼
     ├──► sos-pydantic-author (Sonnet) ──► produces Pydantic model (depends on schema)
     │                                    │
     │                                    ▼
     └──► sos-contract-tester (Haiku) ──► produces test file (depends on schema + model)

  ┌─────────── me ───────────┐
  │ run tests, review diff,  │
  │ commit to staging branch │
  └──────────────────────────┘

If pipe breaks during sprint:
  ├──► sos-medic via bus (tmux wake) — handles incident, logs to EXPERIENCE.md
```

## Dispatch cadence

- Me (opus, coordinator): sends Task tool briefs, reviews outputs, commits to branch
- Parallel dispatch: when outputs are independent (e.g., 3 schema files at once), send all in one message with multiple `Agent` tool calls
- Sequential dispatch: when output B depends on output A (pydantic needs schema first), do in series
- sos-medic never invoked directly from this flow — only if a pipe breaks during sprint
