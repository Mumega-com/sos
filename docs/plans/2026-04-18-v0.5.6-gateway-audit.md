# v0.5.6 — Gateway Bridge Audit-Wrapper

**Date:** 2026-04-18
**Predecessor:** v0.5.5 (squad audit, in-dep).
**Scope:** Emit `POLICY_DECISION` events for every authenticated call
through `sos/services/gateway/bridge.py::require_tenant`. Same in-dep
pattern as v0.5.5 — no route signature changes.

## Context

`gateway/bridge.py` is the public front door for external agents (ChatGPT,
Claude, custom integrations). Its auth is intentionally separate from the
kernel's tokens.json:

- API keys live in `~/.sos/data/gateway/tenants.json` (a flat JSON map
  keyed by API key).
- Keys are minted via `POST /register` — not through `sos.kernel.auth`.
- `MUMEGA_INTERNAL_KEY` env secret bypasses the lookup (returns
  `tenant_id="mumega"`).

Forcing this through `can_execute()` would require registering every
external API key in the kernel auth store, which defeats the purpose of
an isolated external-agent registry. Audit-wrapper is the right call.

## Authenticated routes (6)

All use `tenant_id: str = Depends(require_tenant)`:

- `GET /manifest` — tenant capabilities readback
- `POST /chat` — agent message proxy
- `POST /tasks/create` — task registration
- `POST /memory/store` — memory write
- `POST /memory/search` — memory search
- `GET /system/status` — system health (extra admin-in-tenant check)

Public routes (`GET /`, `POST /register`) are unchanged.

## The pattern

Same shape as `squad/auth.py` v0.5.5 — inject audit emission into the
existing `require_tenant` dependency:

```python
async def _emit_gateway_policy(
    *, agent: str, action: str, target: str, tenant: str,
    allowed: bool, reason: str,
) -> None:
    try:
        from sos.contracts.audit import AuditDecision, AuditEventKind
        from sos.kernel.audit import append_event, new_event
        await append_event(new_event(
            agent=agent, tenant=tenant,
            kind=AuditEventKind.POLICY_DECISION,
            action=action, target=target,
            decision=AuditDecision.ALLOW if allowed else AuditDecision.DENY,
            reason=reason, policy_tier="gateway_bridge",
        ))
    except Exception:
        pass


async def require_tenant(...):
    # ... resolve api_key ...
    if not api_key:
        await _emit_gateway_policy(
            agent="anonymous", action="gateway:request", target="bridge",
            tenant="unknown", allowed=False, reason="missing_api_key",
        )
        raise HTTPException(401, "API key required")

    if internal_key and api_key == internal_key:
        await _emit_gateway_policy(
            agent="internal", action="gateway:request", target="bridge",
            tenant="mumega", allowed=True, reason="internal_key",
        )
        return "mumega"

    tenant = get_tenant_by_key(api_key)
    if not tenant:
        await _emit_gateway_policy(
            agent="anonymous", action="gateway:request", target="bridge",
            tenant="unknown", allowed=False, reason="invalid_api_key",
        )
        raise HTTPException(401, "Invalid API key")

    await _emit_gateway_policy(
        agent=tenant["id"], action="gateway:request", target="bridge",
        tenant=tenant["id"], allowed=True, reason="tenant_key",
    )
    return tenant["id"]
```

**Action name:** the dependency only knows "this is a gateway call";
the specific route (chat vs memory/search vs …) isn't visible inside the
dep. That's fine for v0.5.6 — operators reading audit see one
`gateway:request` per authenticated call, with the `tenant` telling them
*who*, and the access log telling them *where*. If we later want
per-route action names, the fix is a route-aware factory (same pattern
as saas's `audited_admin(action)`) but that's more churn than this
stage warrants.
